# Sinair LLM Bot — Project Journal

## Vision

A bot that uses LLM to seamlessly communicate in a sinair.net group chat with 3-10 people.
Must support: personalities, memory of chat users, chat history awareness.

Chat library: https://github.com/iassasin/wschatapi
Chat constraints: UI shows only last ~100 messages; on room join server re-sends ~100 last messages/events.

## Architecture

- **backend/** — Spring Boot/Kotlin. Stores chat data, will host LLM integration later.
- **chat-collector/** — Node.js service using @iassasin/wschatapi. Connects to chat via WebSocket, forwards messages/events to backend for storage.
- **frontend/** — Next.js (template, not actively used for bot yet).

## Done

- [x] Chat data storage schema (chat_message + chat_event tables)

- [x] Deduplication logic:
  - Messages WITH id: dedup_key = "ext:{id}"
  - Messages WITHOUT id: dedup_key = "hash:{sha256(room|sender|text|time)}"
  - Events: dedup_key = "hash:{sha256(room|member|status|time|data)}"
- [x] Backend ingest endpoint: POST /api/chat/ingest (permitAll for internal collector)
- [x] Chat collector service (chat-collector/) with batching and auto-reconnect
- [x] Chat collector Dockerfile and Docker Compose deployment
- [x] LLM bot architecture (backend `bot/` feature)
  - OpenRouter (OpenAI-compatible chat completions) via Spring RestClient
  - Configurable model tiers (gate/cheap/smart) + app.llm.active-reply-tier switch
  - 3-stage request strategy: heuristic gate (0 tokens) -> interest classifier (gate tier) -> reply generator (cheap/smart)
  - Rolling room summary + presence line (from chat_event) for token-efficient context
  - Per-room cooldown + sliding-window rate limit; optional spontaneous chime-in
  - Persona = `segfault` (configurable), blends into the chat
  - Outbound queue: backend stores replies, collector polls GET /api/chat/outbound, sends via WS, POSTs /ack
  - Migration V100002 (outbound_message, room_summary)
- [x] Per-room quiet-period debounce (judge a burst once after it settles)
- [x] Full-stub local stack: llm-stubs WireMock module + `stub-llm` profile (like google-stubs/stub-google); chat-stubs /nick handling; bot env wired in templates/docker (.env.test, docker-compose.local.yml)
- [x] Automation test: backend BotConversationScenarioTest (few-messages scenario; uses `stub-llm` WireMock; run with `mvn -pl backend -am test -Dtest=BotConversationScenarioTest -Dsurefire.failIfNoSpecifiedTests=false`)

## Notes

- Some messages/events have no IDs — dedup uses composite hash
- Every room join re-sends ~100 last messages — dedup handles this
- typing/stop_typing events are filtered out (no training value)
- Collector batches items (default: 10 or every 5s) before sending to backend
